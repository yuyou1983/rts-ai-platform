# Environment Schema Reference

This document describes the `harness/config/environment.json` file — the environment contract generated by **harness-creator** and consumed by **harness-executor**.

> **Purpose**: `environment.json` describes the complete runtime ecosystem a project needs. It enables the executor to set up services, start the application, and provide context to the **Verifier Subagent** for functional verification.

---

## Relationship with Other Config Files

```
harness/config/
├── validate.json       # Static validation (build, lint, test) → validate.py
├── environment.json    # Runtime environment contract → consumed by executor and Verifier Subagent
└── verify.json         # Optional: standalone smoke checks → verify.py (not part of the skill flow)
```

| File | Created By | Purpose |
|------|------------|---------|
| `validate.json` | harness-creator | Static checks (build, lint, test commands) |
| `environment.json` | harness-creator | Runtime environment: startup, services, env vars |
| `verify.json` | Optional | Standalone smoke check config for `verify.py` tool (not required by the skill flow) |

**Key distinction**: `environment.json` answers "what does this project need to run?" — the Verifier Subagent uses this context to start the application, set up test environments, and execute verification scenarios.

---

## Schema Overview

### Top-Level Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | string | Yes | Schema version ("2.0") |
| `project` | object | Yes | Project name, type, language, description |
| `project_name` | string | Yes | Project identifier (deprecated, use `project.name`) |
| `generated_at` | ISO datetime | Yes | When this was generated |
| `generated_by` | string | Yes | Always "harness-creator" |
| `runtime` | object | Yes | Language, build/dev commands |
| `startup` | object | Yes* | Application startup: command, args, readiness check (* except libraries) |
| `services` | array | No | Service dependencies (databases, caches, queues) |
| `databases` | array | No | Database dependencies (legacy, prefer services[]) |
| `env_vars` | object | No | Required and optional environment variables |
| `secrets` | array | No | Required secrets/credentials (legacy, prefer env_vars.required) |
| `endpoints` | object | No | Known endpoints (health, base_url) |
| `ports` | array | No | Ports the application uses |
| `files` | object | No | Required/generated files |
| `functional_scenarios` | array | No | Functional verification scenarios |
| `test_environment` | object | No | Test-mode environment variables |
| `scripts` | object | No | Paths to lifecycle scripts |
| `_meta` | object | No | Generation metadata, TODOs, user input requirements |

### `project` Object (v2.0)

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Project identifier |
| `type` | string | "web-api", "cli", "frontend", "library", "hybrid" |
| `language` | string | Primary language |
| `description` | string | Brief project description |

### `startup` Object (v2.0 — new)

> **This is the most important section for the executor.** It tells the executor how to start and detect readiness of the application for verification.

| Field | Type | Description |
|-------|------|-------------|
| `command` | string | The command to start the application (e.g., "go run ./cmd/server") |
| `args` | string[] | Optional arguments |
| `working_dir` | string | Working directory (default ".") |
| `env` | object | Environment variables needed for startup |
| `readiness` | object | How to know the app is ready to receive requests |
| `readiness.type` | string | "http", "tcp", "log_pattern", "process", "none" |
| `readiness.config` | object | Type-specific configuration |

**Readiness types:**

| Type | Config Fields | Use When |
|------|--------------|----------|
| `http` | `endpoint`, `port`, `expected_status`, `timeout_seconds`, `poll_interval_ms` | Web API with health endpoint |
| `tcp` | `host`, `port`, `timeout_seconds` | Service that listens on port but no HTTP |
| `log_pattern` | `pattern`, `timeout_seconds` | Service that logs "ready" message |
| `process` | `command` | Just check process is running |
| `none` | — | Library or CLI (no startup needed) |

### `runtime` Object

| Field | Type | Description |
|-------|------|-------------|
| `language` | string | "go", "typescript", "python", "java" |
| `version` | string | Minimum version required |
| `package_manager` | string | "npm", "pnpm", "yarn", "pip", "poetry", "maven", "gradle" |
| `build_command` | string | How to build the project |
| `test_command` | string | How to run tests |
| `lint_command` | string | How to run linting |
| `dev_command` | string | How to start in dev mode (legacy, prefer startup.command) |

### `env_vars` Object (v2.0 — enhanced)

Structured environment variable definitions with metadata.

```json
{
  "env_vars": {
    "required": {
      "DATABASE_URL": {
        "purpose": "PostgreSQL connection string",
        "sensitive": true,
        "example": "postgres://user:pass@localhost:5432/dbname"
      }
    },
    "optional": {
      "PORT": {
        "purpose": "HTTP server port",
        "default": "8080",
        "sensitive": false
      }
    }
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `required` | object | Required env vars — app fails without them |
| `required.<name>.purpose` | string | What this variable is for |
| `required.<name>.sensitive` | boolean | Is this a secret? |
| `required.<name>.example` | string | Example value (for non-sensitive) |
| `required.<name>.test_value_ok` | boolean | Safe to use a test value? |
| `required.<name>.test_value` | string | Safe test value (only if test_value_ok) |
| `optional` | object | Optional env vars with defaults |
| `optional.<name>.default` | string | Default value if not set |

### `databases[]` Array

Each entry describes a database dependency:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Unique identifier (e.g., "primary_db") |
| `type` | string | "postgres", "mysql", "mongodb", "redis", "sqlite" |
| `purpose` | string | What this database is used for |
| `required` | boolean | Whether the app fails without it |
| `connection` | object | Environment variables for connection |
| `connection.host_env` | string | Env var for host |
| `connection.port_env` | string | Env var for port |
| `connection.default_port` | number | Default port if env var not set |
| `connection.user_env` | string | Env var for username |
| `connection.password_env` | string | Env var for password |
| `connection.database_env` | string | Env var for database name |
| `connection.url_env` | string | Env var for full connection URL |
| `setup.docker_image` | string | Docker image to use |
| `setup.docker_compose_service` | string | Service name in docker-compose.yml |
| `setup.migration_command` | string | How to run migrations |
| `setup.seed_command` | string | How to seed test data |
| `test_alternatives` | object | Alternative configs for testing |

### `services[]` Array

Each entry describes an external service dependency:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Unique identifier |
| `type` | string | "redis", "http", "grpc", "kafka", "rabbitmq", "s3" |
| `purpose` | string | What this service is used for |
| `required` | boolean | Whether the app fails without it |
| `connection.url_env` | string | Env var for connection URL |
| `connection.default_url` | string | Default URL |
| `connection.health_endpoint` | string | Health check path (for HTTP services) |
| `setup.docker_image` | string | Docker image |
| `setup.docker_compose_service` | string | docker-compose service name |
| `fallback` | string | What happens when service is unavailable |
| `test_alternatives.mock` | string | How to use a mock instead |

### `secrets[]` Array

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Environment variable name |
| `purpose` | string | What this secret is used for |
| `required` | boolean | Whether the app fails without it |
| `test_value_ok` | boolean | Is it safe to use a test value? |
| `test_value` | string | Safe test value (only if `test_value_ok: true`) |
| `skip_when_missing` | string | Behavior when secret is missing |

### `functional_scenarios[]` Array

These describe **what to functionally verify** — the verifier subagent uses them.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Scenario identifier (e.g., "user_auth_flow") |
| `description` | string | Natural language description |
| `requires` | string[] | Names from databases/services/secrets needed |
| `category` | string | "auth", "core", "infra", "integration" |
| `steps_hint` | string[] | Natural language step hints (not precise specs) |
| `priority` | string | "high", "medium", "low" |

**Important**: `steps_hint` is intentionally natural language, not exact HTTP specs. The verifier subagent reads the code to fill in precise details (request bodies, headers, etc.). This keeps scenarios robust across refactors.

### `test_environment` Object

| Field | Type | Description |
|-------|------|-------------|
| `env_vars` | object | Key-value pairs of safe test environment variables |
| `setup_commands` | string[] | Commands to run before testing |
| `teardown_commands` | string[] | Commands to run after testing |

### `scripts` Object

| Field | Type | Description |
|-------|------|-------------|
| `setup_env` | string | Path to environment setup script |
| `start_server` | string | Path to server start script |
| `teardown_env` | string | Path to environment teardown script |
| `seed_data` | string | Path to data seeding script |

## How the Executor Consumes environment.json

### 1. Verifier Subagent (Functional Verification — PRIMARY)

The executor's primary use of `environment.json` is to provide context to the **Verifier Subagent**:

```
environment.json (startup, readiness, services, env_vars)
    + task context (changed files, new endpoints)
    + coordinator-designed scenarios
    ───────────────────────────────────────────
    → Verifier Subagent starts server, executes scenarios, reports results
```

The Verifier Subagent receives:
- **Startup config**: How to start the app (from environment.json startup section)
- **Readiness check**: How to detect the app is ready (from startup.readiness)
- **Functional scenarios**: Pre-defined scenarios from `functional_scenarios[]`
- **Environment context**: Available databases, services, env vars

The subagent then:
1. Starts the application using `startup.command`
2. Waits for readiness using `startup.readiness` config
3. Sets up the test environment using safe defaults from `env_vars.optional`
4. Executes coordinator-designed task-specific scenarios
5. Executes pre-defined scenarios from `functional_scenarios[]`
6. Verifies functional correctness with real HTTP requests and side-effect checks

### 2. preflight.py (Prerequisite Checking)

Before any verification runs, `preflight.py` loads `environment.json` and checks:

- **services**: Docker containers running, TCP connectivity
- **env_vars.required**: Required env vars are set (not empty)
- **endpoints**: Required ports are available
- **files**: Required files exist

### 3. Coordinator (Environment Setup/Teardown)

The coordinator uses `scripts` paths to manage the environment lifecycle:

```
Setup: harness/scripts/setup-env.sh  → Start DB, Redis, etc.
Start: harness/scripts/start-server.sh → Start the application
Test:  Verifier Subagent runs scenarios with real HTTP requests
Stop:  harness/scripts/teardown-env.sh → Stop everything
```

## Backward Compatibility

- `environment.json` is **optional** — existing projects without it continue to work normally
- When absent, the executor falls back to:
  1. `detect_adapter.py` for language/build commands
  2. Heuristic startup detection (Makefile, Dockerfile, package.json)
- **Functional verification is ALWAYS mandatory** for all tasks, regardless of whether `environment.json` exists. The Coordinator designs task-specific scenarios that work even without environment.json — the Verifier Subagent can use in-memory/mock data.

## Migration from v1.0

Projects with v1.0 `environment.json` (using `runtime.dev_command` instead of `startup.command`) are automatically handled:

| v1.0 Field | v2.0 Equivalent | Auto-Migration |
|------------|-----------------|----------------|
| `runtime.dev_command` | `startup.command` | Yes — executor checks both |
| `databases[]` | `services[]` with `type: "database"` | Yes — backward compatible |
| `secrets[]` | `env_vars.required` with `sensitive: true` | Yes — executor reads both |
| `test_environment.env_vars` | `env_vars.optional` | Yes — executor merges |

Projects with pre-generated `verify.json` from harness-creator can still use it with the standalone `verify.py` tool for quick smoke checks.
