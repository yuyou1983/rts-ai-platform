# verify.py Standalone Tool Guide

> **Note**: This guide documents `verify.py` as a **standalone command-line tool** for quick smoke testing. The harness-executor skill flow uses the **Verifier Subagent** (see `functional-verification-guide.md`) as the primary verification mechanism. `verify.py` can still be run manually for fast sanity checks.

`verify.py` runs deterministic smoke checks: starts servers, checks health endpoints, runs CLI commands, and validates frontend rendering. It's useful for quick local testing before spawning the full Verifier Subagent.

## When to Use verify.py

| Use Case | Tool |
|----------|------|
| Quick local sanity check | `verify.py` (this tool) |
| Skill-integrated verification | Verifier Subagent (see functional-verification-guide.md) |
| Task completion gate | Verifier Subagent (produces verification-report.json) |

## Architecture

```
harness/config/verify.json          ← Configuration (generated by creator, customized by user)
         ↓
verify.py --generate-config         ← Auto-detects app type, creates default config
verify.py .                         ← Runs the full verification pipeline
verify.py . --type server           ← Runs only server verification
         ↓
┌─────────────────────────────────────┐
│    Prerequisites (Preflight Check)   │
│                                     │
│ • Database connectivity             │
│ • Required env vars                 │
│ • External services reachable       │
│ • Custom commands succeed           │
└──────────────┬──────────────────────┘
               │ All pass?
               ↓
┌─────────────────────────────────────┐
│         Verification Pipeline        │
├──────────┬──────────┬───────────────┤
│  Server  │   CLI    │   Frontend    │
│          │          │               │
│ Start →  │ Build →  │ Dev server →  │
│ Ready →  │ Run →    │ Ready →       │
│ Test →   │ Check →  │ CDP test →    │
│ Stop     │          │ Stop          │
└──────────┴──────────┴───────────────┘
         ↓
    Smoke Tests
         ↓
    Cleanup
         ↓
  VerifyReport (JSON)
```

## Prerequisites (Environment Preflight Checks)

**NEW**: Before running any verification, `verify.py` checks that the environment is ready. This prevents wasted time on verification attempts that will fail due to missing database connections, secrets, or services.

### What Prerequisites Check

| Check Type | What It Validates | Example |
|---|---|---|
| `databases` | Database connectivity (MySQL, PostgreSQL, Redis, etc.) | Can connect to `$DB_HOST:$DB_PORT` |
| `env_vars` | Required environment variables exist and have valid values | `JWT_SECRET` is set and non-empty |
| `services` | External services are reachable | Auth service at `http://localhost:9000/health` returns 200 |
| `commands` | Custom commands succeed | `docker info` exits 0 |
| `paths` | Required files/directories exist | `docker-compose.yml` exists |

### Prerequisite Failure Behavior

If any **required** prerequisite fails:
- Verification is **skipped** (not failed)
- A clear error message explains what's missing
- Exit code is 2 (setup failure)

If a **non-required** prerequisite fails:
- A warning is logged
- Verification proceeds

### Example: Spring Boot with MySQL

```json
{
  "prerequisites": {
    "databases": [
      {
        "name": "MySQL Database",
        "type": "mysql",
        "host_env": "DB_HOST",
        "port_env": "DB_PORT",
        "required": true
      }
    ],
    "env_vars": [
      {"name": "DB_HOST", "required": true, "not_empty": true},
      {"name": "DB_PORT", "required": true},
      {"name": "DB_USERNAME", "required": true},
      {"name": "DB_PASSWORD", "required": true},
      {"name": "JWT_SECRET", "required": true}
    ]
  }
}
```

### Skipping Prerequisites

For CI environments without full infrastructure:

```json
{
  "prerequisites": {
    "skip": true
  }
}
```

Or via command line:

```bash
verify.py . --skip-prerequisites
```

## Configuration

### Config Location

`harness/config/verify.json` — created by harness-creator or auto-generated by verify.py on first run.

### How Config is Created

1. **harness-creator** analyzes the project and generates `verify.json` during Create or Greenfield mode
2. If no config exists, **verify.py** auto-detects the app type and generates a default config
3. Users can manually edit the config to add endpoints, CLI commands, or page tests

### Auto-Detection Heuristics

verify.py determines the app type by scoring indicators:

**Server indicators:**
- `cmd/server/` directory exists (+3)
- `http.ListenAndServe` in Go source (+2)
- Express/Fastify/Koa/Hono in package.json (+3)
- FastAPI/Flask in Python source (+3)

**CLI indicators:**
- `cmd/cli/` directory exists (+3)
- Cobra/urfave/cli imports (+2)
- Commander/yargs in package.json (+2)

**Frontend indicators:**
- React/Vue/Svelte/Next/Vite in package.json (+3)
- Vite/Next in dev script (+2)

If multiple types score high, the app is "hybrid" and all relevant verification types run.

## Server Verification

### How It Works

1. **Start** — Launches the server as a background process
2. **Readiness** — Polls until the server is ready (HTTP health check, TCP port, or log pattern)
3. **Endpoint Tests** — Sends HTTP requests and checks responses
4. **Stop** — Gracefully shuts down the server (SIGTERM → wait → SIGKILL if needed)

### Readiness Strategies

| Strategy | When to Use | Configuration |
|---|---|---|
| `http` | Server has a health endpoint | `endpoint`, `expected_status` |
| `tcp` | No health endpoint, but known port | `host`, `port` |
| `log_pattern` | Server logs "ready" message | `pattern` (regex) |
| `command` | Custom readiness check | `command` (exit 0 = ready) |

### Endpoint Testing

Each endpoint test specifies:
- **Method + Path** — What to call
- **Headers + Body** — What to send
- **Expected response** — Status code, body content, JSON path assertions

JSON path assertions use a simplified syntax:
```json
{"path": "$.data.id", "operator": "exists"}
{"path": "$.data.name", "operator": "equals", "value": "test"}
{"path": "$.items", "operator": "type", "value": "array"}
```

### Example: Verifying a Go API Server

```json
{
  "server": {
    "start": {
      "command": "go run cmd/server/main.go",
      "env": {"PORT": "8081", "ENV": "test", "DB_URL": "sqlite3://:memory:"},
      "background": true
    },
    "readiness": {
      "type": "http",
      "endpoint": "http://localhost:8081/health",
      "timeout_seconds": 30
    },
    "endpoints": [
      {"name": "health", "method": "GET", "path": "/health", "expected": {"status": 200}},
      {
        "name": "list users",
        "method": "GET",
        "path": "/api/v1/users",
        "expected": {"status": 200, "json_path": [{"path": "$.data", "operator": "type", "value": "array"}]}
      }
    ],
    "stop": {"signal": "SIGTERM", "graceful_timeout_seconds": 5}
  }
}
```

## CLI Verification

### How It Works

1. **Build** — Optionally builds the CLI binary first
2. **Command Tests** — Runs CLI commands and checks exit codes, stdout, stderr, and file outputs

### What to Test

Focus on commands that exercise core functionality:
- `--version` and `--help` — Basic sanity
- Core commands with typical inputs
- Error cases (bad input → non-zero exit code)
- File output commands (verify files are created with expected content)

### Example: Verifying a CLI Tool

```json
{
  "cli": {
    "binary": {
      "build_command": "go build -o bin/mycli cmd/cli/main.go",
      "path": "bin/mycli"
    },
    "commands": [
      {"name": "version", "args": ["--version"], "expected": {"exit_code": 0, "stdout_matches": "v\\d+\\.\\d+"}},
      {"name": "help", "args": ["--help"], "expected": {"exit_code": 0, "stdout_contains": ["Usage", "Commands"]}},
      {
        "name": "process file",
        "args": ["process", "--input", "testdata/sample.json", "--output", "/tmp/result.json"],
        "expected": {
          "exit_code": 0,
          "file_created": ["/tmp/result.json"],
          "file_content": [{"path": "/tmp/result.json", "contains": ["processed"]}]
        }
      },
      {"name": "bad input", "args": ["process", "--input", "nonexistent.json"], "expected": {"exit_code": 1}}
    ]
  }
}
```

## Frontend Verification (CDP)

### How It Works

1. **Dev Server** — Starts the frontend dev server as a background process
2. **Readiness** — Polls until the dev server responds
3. **Page Tests** — Uses headless Chrome to load pages, run actions, and check assertions
4. **Screenshots** — Optionally captures screenshots (useful for debugging failures)

### Chrome DevTools Protocol

verify.py uses headless Chrome in its simplified mode. For full CDP interaction (clicking buttons, filling forms), it launches Chrome with `--headless` and `--screenshot` flags.

For more sophisticated testing (multi-step user flows, SPA navigation), consider:
- Adding Playwright/Puppeteer as a project dependency
- Configuring the test command in `verify.json` to run those tests

### Page Assertions

| Assertion Type | What It Checks |
|---|---|
| `element_exists` | DOM element matching selector exists |
| `element_text` | Element's text content matches |
| `element_visible` | Element is visible (not hidden) |
| `no_console_errors` | No console.error() calls |
| `no_network_errors` | No failed network requests |
| `title` | Page title matches |
| `url` | Current URL matches |
| `evaluate` | Custom JS expression returns true |

### Example: Verifying a React App

```json
{
  "frontend": {
    "dev_server": {
      "command": "npm run dev",
      "env": {"PORT": "3001"},
      "background": true
    },
    "readiness": {"type": "http", "url": "http://localhost:3001", "timeout_seconds": 60},
    "browser": {"headless": true},
    "pages": [
      {
        "name": "homepage",
        "url": "/",
        "wait_for": {"type": "selector", "selector": "#root"},
        "assertions": [
          {"type": "element_exists", "selector": "#root"},
          {"type": "no_console_errors"},
          {"type": "title", "expected": "My App"}
        ]
      },
      {
        "name": "dashboard after login",
        "url": "/login",
        "actions": [
          {"type": "type", "selector": "#email", "value": "admin@test.com"},
          {"type": "type", "selector": "#password", "value": "test123"},
          {"type": "click", "selector": "button[type=submit]"},
          {"type": "wait", "timeout_ms": 3000}
        ],
        "assertions": [
          {"type": "url", "expected": "/dashboard"},
          {"type": "element_visible", "selector": ".stats-panel"}
        ],
        "screenshot": {"path": "harness/screenshots/dashboard.png"}
      }
    ]
  }
}
```

## Standalone Usage

> **Reminder**: This tool is **not** part of the harness-executor skill flow. The skill uses the Verifier Subagent for functional verification. Use `verify.py` for quick manual sanity checks.

### Commands

```bash
# Full runtime verification (auto-detects or uses verify.json)
python3 $SKILL_DIR/scripts/verify.py .

# JSON output for programmatic parsing
python3 $SKILL_DIR/scripts/verify.py . --json

# Specific type only
python3 $SKILL_DIR/scripts/verify.py . --type server

# Verbose with debug output
python3 $SKILL_DIR/scripts/verify.py . -v

# Generate config without running
python3 $SKILL_DIR/scripts/verify.py . --generate-config

# Save report
python3 $SKILL_DIR/scripts/verify.py . --json --output harness/trace/verify-report.json
```

### Exit Codes

| Code | Meaning |
|---|---|
| 0 | All verifications passed |
| 1 | One or more required verifications failed |
| 2 | Configuration error or setup failure |

## Best Practices

### Writing Good Endpoint Tests

- Test the endpoints your change actually affects
- Include both happy path (200) and error path (400/404) tests
- Use JSON path assertions for structured responses
- Keep timeouts reasonable (10s for most, longer for heavy computation)

### Writing Good CLI Tests

- Always test `--version` and `--help` as a baseline
- Test with real-world-like inputs, not just trivial cases
- Check both stdout content and exit codes
- Clean up any files created by test commands

### Writing Good Page Tests

- Start with "page loads without console errors" as a baseline
- Test the critical user flows, not every possible interaction
- Use stable selectors (IDs, data attributes) not fragile ones (nth-child, class names)
- Screenshots on failure help debug issues

### When to Skip Runtime Verification

- **Library projects** — No server, CLI, or frontend to verify. Static tests are sufficient.
- **Pure refactoring** — If the change is purely internal (no behavioral change), runtime verification may not be needed. But if the project has verify.json configured, run it anyway — it's cheap and catches regressions.
- **Documentation changes** — No code to verify.

## Troubleshooting

### Server won't start
- Check if port is already in use: `lsof -i :8080`
- Check environment variables: are DB credentials needed?
- Look at server stdout in verbose mode (`-v`)

### Chrome not found
- Install Chrome or Chromium
- Set `browser.executable` in verify.json to the correct path
- On CI, use `--no-sandbox` flag

### Timeout waiting for readiness
- Increase `timeout_seconds`
- Verify the server actually starts by running the command manually
- Check if the health endpoint path is correct

### CLI build fails
- Run the build command manually to see full error
- Check if build dependencies are installed
